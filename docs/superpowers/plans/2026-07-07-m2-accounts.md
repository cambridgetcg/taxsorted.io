# M2-Accounts Implementation Plan — Passkey Accounts (plan C)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Real user accounts — sign in with a passkey, sign out, revoke — replacing "anonymous forever" as the ceiling, unlocking the production-filing precondition (RUNBOOK) and the Gov-Client-Multi-Factor fraud header.

**Architecture:** An **account is a set of passkeys, nothing else** — no email, no username, no password, no PII. Signed-in state lives as four new columns on the existing `sessions` row (one cookie, `ts_session`, byte-identical attributes; `ts_device` untouched). Entities get exactly one owner — the anonymous browser session that made them, or an account once claimed; claiming clears `session_id` so session deletion can never cascade away claimed data. WebAuthn ceremonies run on the page origin (taxsorted.io) against rpID `taxsorted.io`; the api (api.taxsorted.io) verifies via `@simplewebauthn/server`. A UV-required assertion on this session is what makes Gov-Client-Multi-Factor honest.

**Tech Stack:** existing (Hono api, engine, Next static front) + `@simplewebauthn/server` (api) and `@simplewebauthn/browser` (frontend), pinned exact — the ONLY new deps.

**Design provenance:** 5-map ground-truth sweep + adversarial two-design panel (security-first vs simplicity-first) + judge synthesis, 2026-07-07. Chassis = simplicity-first; grafted from security-first: explicit (not automatic) adoption at login, restricted recovery sessions, passkey revocation cascades to its sessions, enforced production gates.

## Global Constraints

- **No PII, ever.** `users` = opaque WebAuthn handle + picker label only. No email in v1 (spec §3's "email fallback" consciously narrowed to recovery codes — Yu ratification pending, M3 gate named).
- **Anonymous flows stay byte-identical.** M1 prep tools and sandbox filing keep working with no account; only production connect/submit gates on a passkey session (`403 account_needed`). Sandbox stays fully anonymous-capable.
- **Cookie discipline:** `ts_session` attrs unchanged (httpOnly, Secure-in-prod, SameSite=Lax — load-bearing for the HMRC OAuth callback — host-only, path=/, 1yr rolling). No new cookie. `ts_device` stays the sole Gov-Client-Device-ID source.
- **WebAuthn params, fixed:** `rpID` `taxsorted.io` / origin `https://taxsorted.io` (prod), `localhost` / `http://localhost:3000` (dev) — two new `config.ts` fields, env-at-load, NOT in `assertBootConfig`, deliberately NOT derived from corsOrigins. `userVerification:'required'` + `requireUserVerification:true` both ceremonies; `residentKey:'required'`; `attestationType:'none'`.
- **Session rotation at every sign-in** (register-finish, login-finish, recover): new row → re-parent old row's anonymous entities → delete old row → set cookie. Fixation defeated; statement order makes races fail loud.
- **Tests: no Postgres, no network** — house pattern (fake template-tag sql string-matched to exact statements + `vi.stubEnv` + dynamic import + `doMock('../../db.js')`; software-authenticator helper for ceremonies + ONE frozen real-browser fixture). Existing tests stay green and unmodified (they mock `session.js`/`ownedEntity` wholesale).
- **Plain-words error vocabulary** (the full set): `signed_out` 401, `unknown_passkey` 401, `bad_recovery_code` 401, `ceremony_failed` 422, `challenge_expired` 422, `last_passkey` 422, `passkey_already_registered` 409, `bad_origin` 403, `passkey_needed` 403, `account_needed` 403.
- Account-page copy: hardcoded plain English + `// i18n: deferred to M2` (cockpit convention; do not touch the 6-locale dictionaries).
- All suites green (engine + api + frontend, all workspaces) + typecheck before every commit. Commits end with: `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.

## Identity & session model (the contract every task reads)

- **Middleware context** after Task 4: `sessionId`, `deviceId` (as today) + `accountId` (`user_id` when signed in within a **30-day window** computed from `signed_in_at` — the account-management identity), `userId` (same value but ONLY when `mfa_at` is set — the data identity), `mfaAt`, `mfaFactorRef`. Recovery sign-ins set `accountId` but never `userId` — restriction is emergent: no data access (ownedEntity sees null `userId`), no MFA header; only their forward path (add a passkey, logout) is open.
- **Sign-out** = clear the four columns on this session; **sign out everywhere** = same UPDATE `where user_id=$uid`.
- **Window lapse** (>30 days): session behaves anonymous again, row untouched, frontend shows "sign in again". Caps the MFA timestamp ≤30 days (inside HMRC's ~3-month staleness line, approval.md).
- **Adoption**: account **creation** auto-adopts this browser's anonymous entities (same person the anonymous model already trusted). **Login never auto-adopts** — anonymous entities follow the rotated session, show alongside account entities via the dual-owner predicate, and an explicit adopt door claims them.
- **MFA header decision:** sent iff signed in AND `mfa_at` set. `type=OTHER`; `timestamp=mfa_at`; `unique-reference=sha256hex('taxsorted-mfa-v1:'+credentialId)` — fixed salt forever, precomputed at sign-in as `mfa_factor_ref` (header assembly stays a pure read; doubles as the revocation join key). `Gov-Client-User-IDs` becomes `[['taxsorted', accountId ?? sessionId]]`. Server-asserted only — never via CLIENT_ALLOWLIST.

## Migration — `api/migrations/005_accounts.sql` (final, verbatim)

```sql
-- 005_accounts.sql — passkey accounts (plan C).
-- Plain words: an account is a set of passkeys, nothing else — no email, no
-- password, no personal details. Entities belong to the browser session that
-- made them until they are claimed into an account; claiming clears
-- session_id, so deleting a session row can never cascade away claimed
-- entities or their receipts. Exactly one owner, always.

create table users (
  id uuid primary key default gen_random_uuid(),
  -- Opaque base64url handle authenticators store; reused for every passkey
  -- this account adds so devices overwrite instead of duplicating. No PII.
  webauthn_user_id text not null unique,
  -- Only what the passkey picker shows. Not unique, not an identifier.
  name text not null default 'TaxSorted user',
  created_at timestamptz not null default now()
);

create table passkeys (
  -- The credential id, base64url — the lookup key at sign-in.
  id text primary key,
  user_id uuid not null references users(id) on delete cascade,
  public_key bytea not null,
  -- Signature counter. The library hard-fails a regression only when the
  -- stored counter is above zero (always-zero providers like iCloud never
  -- trip it) — that is the clone signal, kept deliberately.
  counter bigint not null default 0,
  transports text[],
  nickname text,
  created_at timestamptz not null default now(),
  last_used_at timestamptz
);
create index passkeys_user_idx on passkeys(user_id);

create table recovery_codes (
  -- sha256 hex of a single-use ~128-bit random code. High entropy makes a
  -- fast hash safe (random tokens, not human passwords). Shown exactly once.
  code_hash text primary key,
  user_id uuid not null references users(id) on delete cascade,
  used_at timestamptz,
  created_at timestamptz not null default now()
);
create index recovery_codes_user_idx on recovery_codes(user_id);

-- One in-flight WebAuthn challenge per browser session: short-lived,
-- single-use (consumed atomically with delete ... returning at verify).
-- For a brand-new account the pending handle + name live here, so no user
-- row exists until a ceremony actually succeeds.
create table webauthn_challenges (
  session_id uuid primary key references sessions(id) on delete cascade,
  challenge text not null,
  kind text not null check (kind in ('registration', 'authentication')),
  webauthn_user_id text,
  user_name text,
  expires_at timestamptz not null
);

-- A session can now be signed in. Signing out clears these columns.
alter table sessions add column user_id uuid references users(id) on delete cascade;
alter table sessions add column signed_in_at timestamptz;
-- Time of the last successful passkey prompt on THIS session — feeds the
-- Gov-Client-Multi-Factor timestamp. Never set by recovery-code sign-ins.
alter table sessions add column mfa_at timestamptz;
-- Salted-and-hashed credential id of the passkey that asserted — feeds the
-- header's unique-reference AND lets deleting that passkey sign out every
-- session it opened. Same hash function, forever.
alter table sessions add column mfa_factor_ref text;
create index sessions_user_idx on sessions(user_id);
create index sessions_mfa_ref_idx on sessions(mfa_factor_ref);

-- Entities get exactly one owner: a session (anonymous) or a user (claimed).
alter table entities alter column session_id drop not null;
alter table entities add column user_id uuid references users(id) on delete cascade;
alter table entities add constraint entities_one_owner
  check (session_id is not null or user_id is not null);
create index entities_user_idx on entities(user_id);
```

Purely additive for existing rows; the boot runner picks it up by filename.

## Endpoints — `api/src/routes/account.ts`, mounted `app.route('/v1/account', account)` after session middleware

Per-file guard (repo convention): every POST/DELETE requires `Origin` ∈ `config.corsOrigins`, else `403 bad_origin` ("This request came from somewhere we don't recognise.") — closes the subdomain-CSRF gap SameSite=Lax leaves. Zod at every boundary. Markers: **[signed-in]** = `accountId` required (401 `signed_out`); **[passkey]** = `userId` required (403 `passkey_needed`, "Add a passkey to open this door.").

1. `GET /v1/account` — anonymous OK → `{signedIn:false}` | `{signedIn:true, account:{id,name,createdAt}, mfa:boolean, passkeys:[{id,nickname,createdAt,lastUsedAt}], recoveryCodesLeft:n, claimableEntities:n}` (`mfa:false` after recovery sign-in; `claimableEntities` = this browser's anonymous entities).
2. `POST /v1/account/passkey/register/start` body `{name?≤64}` → `PublicKeyCredentialCreationOptionsJSON` (`excludeCredentials` = account's credential ids when signed in; new-account stashes fresh handle+name on the challenge row — no user row until a ceremony succeeds; challenge upsert `kind='registration'` now()+5min; opportunistic GC of expired rows).
3. `POST /v1/account/passkey/register/finish` body `{response, nickname?≤64}` — atomic challenge consume (`delete … returning`) → `verifyRegistrationResponse`. New account: insert user+passkey, **auto-adopt** (`update entities set user_id=$u, session_id=null where session_id=$sid`), mint 10 codes, rotate session → `{signedIn:true, account, adoptedEntities:n, recoveryCodes:[×10]}`. Add-passkey ([passkey]): insert → `{signedIn:true, passkey}`. Recovery-upgrade (recovery session): insert + rotate to FULL session (ceremony IS a fresh UV assertion) → `{signedIn:true, passkey, mfa:true}`. Errors: 422 `challenge_expired`/`ceremony_failed` (detail server-log only), 409 `passkey_already_registered`.
4. `POST /v1/account/login/start` body `{}` → `PublicKeyCredentialRequestOptionsJSON` (UV required, `allowCredentials:[]` — usernameless; feeds conditional UI).
5. `POST /v1/account/login/finish` body `{response}` — consume → lookup `passkeys` by `response.id` → `verifyAuthenticationResponse` (library-default counter hard-fail kept) → persist counter+last_used_at → rotate (mfa_at=now, mfa_factor_ref) — **no adopt**; anonymous entities follow the session → `{signedIn:true, account, claimableEntities:n}`. 401 `unknown_passkey` ("We don't recognise that passkey. Lost them all? Use a recovery code.").
6. `POST /v1/account/adopt` — [passkey] → `update entities set user_id=$u, session_id=null where session_id=$sid and user_id is null` → `{adopted:n}` (idempotent).
7. `POST /v1/account/recover` body `{code}` (normalise: strip spaces/dashes, lowercase) — atomic single-use (`update recovery_codes set used_at=now() where code_hash=$h and used_at is null returning user_id`) → rotate with **mfa_at NULL** → `{signedIn:true, mfa:false, addPasskeyNow:true, recoveryCodesLeft:n}` | 401 `bad_recovery_code` ("That code didn't match. Each code works exactly once."). Recovery session reaches ONLY: GET /v1/account, register start/finish, logout.
8. `POST /v1/account/logout` body `{everywhere?}` — [signed-in], recovery INCLUDED. This browser: clear 4 columns on `$sid`; everywhere: `where user_id=$uid` → `{signedOut:'this-browser'|'everywhere'}` (idempotent).
9. `DELETE /v1/account/passkey/:credentialId` — [passkey]. Last one → 422 `last_passkey` ("This is your only passkey. Add another before removing it."). Else sign out its sessions (`where mfa_factor_ref=$ref`) then delete → `{ok:true, sessionsSignedOut:n}`. Unknown/not-yours → 404 `no_such_door`.
10. `POST /v1/account/recovery-codes` — [passkey] regenerate: delete unused, mint 10 → `{recoveryCodes:[×10]}`.

**Changed existing code:** `session.ts` (RETURNING grows, window, context vars); `ownedEntity` → `where id=$1 and (session_id=$sid or user_id=$uid)` (null `$uid` never matches — anonymous byte-identical); `entities.ts` list dual predicate + create sets `user_id, session_id=null` when passkey-authed; `connect.ts` start + `vat.ts`/`itsa-submit.ts` submit: production env without `userId` → 403 `account_needed`; `fraud.ts` per the MFA contract.
