# M1 — ITSA Engine Module Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `@taxsorted/engine/uk/itsa` — a pure-TS, fully-cited MTD Income Tax module: 2026-27 rates, MTD categories, quarter/deadline logic, cumulative aggregation, eligibility, simplified expenses, a labelled liability estimate, and honest optimisers.

**Architecture:** Mirror the existing `engine/jurisdictions/uk/vat` module shape (config/types/compute/explain/index + `__tests__`). Pure functions, no IO, no DOM. **Money is integer pence** everywhere inside the engine (`type Pence = number`); conversion to HMRC's decimal pounds happens at the API boundary in M2, not here. Every configured value carries `{ value, source, effectiveFrom }` so the UI can cite gov.uk/SI per number (the explain layer IS the citation layer, spec §3/§6).

**Tech Stack:** TypeScript 5, vitest 4 (workspace `engine`), zero new runtime deps.

## Global Constraints

- Facts (rates/dates/thresholds) come from `regs/research/substance.md` + `regs/research/mandate.md` — every constant in `config.ts` must carry the source URL given there. No un-cited numbers.
- 2026-27, rUK (England/Wales/NI) bands only; the estimator output carries `scope: 'rUK'` and `kind: 'estimate'` labels (Scotland deferred; HMRC's calculation is the authoritative number — spec §4).
- Estimator covers non-savings income only (SE + property + other-employment stub). Dividends/savings: out (YAGNI for M1 prep tool; M2 adds them with the HMRC calc side-by-side).
- Property API field names are modelled as our internal keys pending Property Business v6.0 OAS re-verification (research caveat) — keep all wire names in `categories.ts` only, so a rename is a one-file change.
- Tests: `npm test --workspace engine`. All existing VAT/hmrc tests stay green.

```ts
export interface CumulativeUpdate {
  source: SourceType
  quarter: Quarter
  totals: Record<string, Pence>   // categoryKey -> cumulative year-to-date total
  recordCount: number
}
export function cumulativeUpdate(records: LedgerRecord[], source: SourceType, taxYear: TaxYear,
  quarterIndex: 1|2|3|4, opts?: { election?: 'standard'|'calendar'; consolidated?: boolean }): CumulativeUpdate
```

Consolidation folds every `consolidatable !== false && !alwaysSeparate` expense into `consolidatedExpenses`; income categories and `alwaysSeparate` fields keep their own keys. Records outside the cumulative window (year start → quarter end) or for the other source are excluded. Unknown category keys throw.
