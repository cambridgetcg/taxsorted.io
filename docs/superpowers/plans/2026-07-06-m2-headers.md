**Goal:** All 16 WEB_APP_VIA_SERVER fraud-prevention headers complete and passing HMRC's Test Fraud Prevention Headers API — as a repeatable CI job. This is the known stall point of production approval (research: "header sign-off is where approvals stall"); we green it months early.

**Architecture:** The split pipeline exists (`engine/jurisdictions/uk/hmrc/fraud-headers.ts` browser-collect spec + `api/src/fraud.ts` server assembly). Missing per the repo analysis: `Gov-Client-Public-Port`, `Gov-Client-Multi-Factor`, `Gov-Vendor-License-IDs`, `GOV_VENDOR_PUBLIC_IP` env (Vendor-Forwarded currently malformed as `by=&for=<ip>`), vendor version mismatch (api 0.1.0 vs engine 1.0.0). Validation via HMRC's Test Fraud Prevention Headers API (application-restricted = client-credentials — runs headlessly in CI).

**Ground truth:** `regs/research/fraud-headers.md` — the spec v3.3 per-header formats, the five cannot-collect cases (email SDSTeam BEFORE omitting), percent-encoding rules. Every header change cites its section.

## Global Constraints

- Header VALUES must be truthful — a fabricated value (fake MFA claim, made-up license ID) is worse than an honest omission with the documented justification. Cannot-collect cases: `Gov-Client-Multi-Factor` (we have no MFA yet → EMPTY is wrong; the spec's cannot-collect route applies — document, and the value ships once accounts+MFA exist in plan C), `Gov-Vendor-License-IDs` (no licensed software → spec-recognised omission case). Record both in the runbook as SDSTeam-notification items for M3.
- Encoding: US-ASCII, RFC 3986 percent-encoding of keys/values, never separators (research §encoding). Timestamps `yyyy-MM-ddThh:mm:ss.sssZ` (milliseconds mandatory).
- No privacy expansion beyond the legal requirement: collect only what the spec names, disclosure page updated in the same wave.
- All suites green; commits end with: `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
